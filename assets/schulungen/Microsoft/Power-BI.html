Desktop vs. Cloud – Wo passiert was?

Power BI hat zwei Hauptkomponenten die sich ergänzen. Desktop ist das Entwicklungswerkzeug, der Service ist die Distributions- und Kollaborationsplattform.

🖥️

Power BI Desktop

Kostenlos – lokale Windows-Anwendung. Hier entstehen Reports: Daten laden, transformieren (Power Query), Datenmodell aufbauen, DAX-Measures schreiben, Visuals gestalten.

☁️

Power BI Service (Cloud)

Pro/Premium-Lizenz – app.powerbi.com. Reports veröffentlichen, teilen, automatisch aktualisieren (Scheduled Refresh), Dashboards pinnen, Dataflows zentral verwalten.

DAX – Zeitintelligenz

Zeitintelligenz-Funktionen sind das Herzstück jeder Business-Intelligence-Lösung. Sie erlauben Vergleiche zwischen Perioden ohne manuelle Filterlogik.

⚠️
Voraussetzung: Eine vollständige Datumstabelle muss im Modell als Datumstabelle markiert sein (Tabellentools → Als Datumstabelle markieren). Ohne das funktionieren Zeitintelligenz-Funktionen nicht korrekt.
Datumstabelle erstellen (DAX)
DAX – Calculated Table 
-- Vollständige Datumstabelle als berechnete Tabelle
Datumstabelle =
VAR StartDatum = DATE(2020, 1, 1)
VAR EndDatum   = DATE(2026, 12, 31)
RETURN
ADDCOLUMNS(
    CALENDAR(StartDatum, EndDatum),
    "Jahr",          YEAR([Date]),
    "Quartal",       "Q" & QUARTER([Date]),
    "Monat Nr",      MONTH([Date]),
    "Monat Name",    FORMAT([Date], "MMMM"),
    "Monat Kurz",    FORMAT([Date], "MMM"),
    "KW",            WEEKNUM([Date], 2),
    "Wochentag",     FORMAT([Date], "DDDD"),
    "Jahr-Monat",    FORMAT([Date], "YYYY-MM"),
    "Ist Wochenende", IF(WEEKDAY([Date],2) >= 6, "Ja", "Nein")
)
Zeitintelligenz-Measures
Measure-Name DAX-Formel Beschreibung
Umsatz VJ CALCULATE([Umsatz], SAMEPERIODLASTYEAR(Datum[Date])) Gleicher Zeitraum Vorjahr. Automatischer Jahresversatz.
YTD Umsatz CALCULATE([Umsatz], DATESYTD(Datum[Date])) Kumulierter Umsatz vom 1. Januar bis zum aktuellen Datum.
YTD VJ CALCULATE([Umsatz], DATESYTD(SAMEPERIODLASTYEAR(Datum[Date]))) Kumuliert Vorjahr – ideal für YTD-Vergleich.
Letzte 12 Monate CALCULATE([Umsatz], DATESINPERIOD(Datum[Date], LASTDATE(Datum[Date]), -12, MONTH)) Rolling 12 Months – bewegt sich mit dem letzten Datum.
Wachstum % VJ DIVIDE([Umsatz] - [Umsatz VJ], [Umsatz VJ], BLANK()) Prozentuales Wachstum vs. Vorjahr. DIVIDE verhindert Division durch 0.
Vormonat CALCULATE([Umsatz], DATEADD(Datum[Date], -1, MONTH)) Flexibler Zeitversatz: -1 MONTH, -1 QUARTER, -1 YEAR etc.
MTD CALCULATE([Umsatz], DATESMTD(Datum[Date])) Month-to-Date: kumuliert seit Monatsbeginn.
QTD CALCULATE([Umsatz], DATESQTD(Datum[Date])) Quarter-to-Date: kumuliert seit Quartalsbeginn.
✅
Best Practice: Schreibe alle Zeitintelligenz-Measures auf Basis eines Basis-Measures (z.B. [Umsatz]). Ändere die Basisberechnung einmal, alle Zeitvergleiche folgen automatisch.

DAX – CALCULATE & Filtermodifikatoren

CALCULATE ist die mächtigste DAX-Funktion. Sie verändert den Filterkontext – das ist die Grundlage fast jeder komplexen Berechnung in Power BI.

DAX – CALCULATE Grundprinzip 
-- CALCULATE(Ausdruck, Filter1, Filter2, ...)
-- Filter ERSETZEN den bestehenden Filterkontext

-- Beispiel 1: Umsatz nur für Kategorie "Elektronik"
Umsatz Elektronik =
CALCULATE([Umsatz], Produkt[Kategorie] = "Elektronik")

-- Beispiel 2: ALL() – alle Filter einer Spalte entfernen
Marktanteil % =
DIVIDE(
    [Umsatz],
    CALCULATE([Umsatz], ALL(Produkt[Kategorie])),
    0
)

-- Beispiel 3: ALLEXCEPT() – alle Filter bis auf eine Spalte entfernen
Umsatz je Region (gesamt) =
CALCULATE([Umsatz], ALLEXCEPT(Vertrieb, Vertrieb[Region]))

-- Beispiel 4: KEEPFILTERS() – Filter hinzufügen statt ersetzen
Umsatz Top-Kunden =
CALCULATE([Umsatz], KEEPFILTERS(Kunde[Segment] = "Premium"))

-- Beispiel 5: REMOVEFILTERS() – gezielt Filters entfernen (neuere Syntax)
Gesamtumsatz (immer) =
CALCULATE([Umsatz], REMOVEFILTERS(Produkt))
🗑️

ALL()

Entfernt alle Filter von einer Tabelle oder Spalte. Basis für Anteils- und Ranking-Berechnungen.

🎯

KEEPFILTERS()

Filter werden ADDIERT statt ersetzt. Nützlich wenn der Reportfilter erhalten bleiben soll.

🔄

USERELATIONSHIP()

Aktiviert eine inaktive Beziehung temporär für die Berechnung (z.B. Lieferdatum vs. Bestelldatum).

DAX – USERELATIONSHIP & CROSSFILTER 
-- Zwei Datumsspalten mit einer Datumstabelle verknüpfen
-- Aktive Beziehung: Bestelldatum | Inaktive: Lieferdatum

Umsatz nach Lieferdatum =
CALCULATE(
    [Umsatz],
    USERELATIONSHIP(Datum[Date], Bestellung[Lieferdatum])
)

-- CROSSFILTER: Filterrichtung für eine Berechnung ändern
Kunden mit Bestellungen =
CALCULATE(
    DISTINCTCOUNT(Bestellung[KundenID]),
    CROSSFILTER(Kunde[ID], Bestellung[KundenID], Both)
)

DAX – Iteratoren & VAR/RETURN

X-Funktionen (SUMX, AVERAGEX, RANKX ...) iterieren zeilenweise und öffnen riesige Möglichkeiten. VAR/RETURN macht komplexe Formeln lesbar und performant.

DAX – Iteratoren (X-Functions) 
-- SUMX: berechnet Ausdruck PRO ZEILE, dann summiert
Umsatz = SUMX(Bestellung, Bestellung[Menge] * Bestellung[Preis])

-- AVERAGEX: Durchschnitt über berechnete Werte
Ø Bestellwert = AVERAGEX(Bestellung, Bestellung[Menge] * Bestellung[Preis])

-- RANKX: Ranking innerhalb einer Gruppe
Produktrang =
RANKX(
    ALL(Produkt[Produktname]),   -- Alle Produkte (Filter entfernen)
    [Umsatz],                        -- Nach diesem Wert ranken
    ,                                -- Kein spezifischer Wert → aktueller Kontext
    DESC,                            -- Absteigend (höchster Umsatz = Rang 1)
    Dense                            -- Gleiche Werte = gleicher Rang, kein Skip
)

-- MAXX / MINX: Maximum/Minimum über berechneten Ausdruck
Größte Einzelbestellung =
MAXX(Bestellung, Bestellung[Menge] * Bestellung[Preis])
DAX – VAR/RETURN Pattern (Best Practice) 
-- VAR: Variablen deklarieren → einmal berechnen, mehrfach nutzen
-- Vorteil: lesbarer Code + bessere Performance (kein doppeltes Auswerten)

Wachstum mit Ampel =
VAR AktuellerUmsatz  = [Umsatz]
VAR VorjahrUmsatz    = [Umsatz VJ]
VAR Wachstumsrate    = DIVIDE(AktuellerUmsatz - VorjahrUmsatz, VorjahrUmsatz)
RETURN
SWITCH(
    TRUE(),
    ISBLANK(VorjahrUmsatz),          "Kein VJ",
    Wachstumsrate >= 0.1,              "🟢 +" & FORMAT(Wachstumsrate, "0.0%"),
    Wachstumsrate >= 0,                "🟡 +" & FORMAT(Wachstumsrate, "0.0%"),
    "🔴 " & FORMAT(Wachstumsrate, "0.0%")
)

-- Komplexes Beispiel: Dynamischer Top-N Filter
Top-N Umsatz =
VAR N            = SELECTEDVALUE(TopN[Wert], 10)
VAR TopProdukte   = TOPN(N, ALL(Produkt[Name]), [Umsatz], DESC)
VAR IstImTop      = ISINSCOPE(Produkt[Name]) && Produkt[Name] IN TopProdukte
RETURN
IF(IstImTop, [Umsatz])

Datenmodellierung – Star Schema

Ein gutes Datenmodell ist wichtiger als perfekte DAX-Formeln. Das Star Schema ist der Standard für Power BI.

⭐

Faktentabelle

Enthält Transaktionen/Messungen (Verkäufe, Buchungen, Klicks). Hat viele Zeilen, wenig Spalten. Enthält Fremdschlüssel zu Dimensionen und Messwerte (Menge, Preis, Betrag).

📋

Dimensionstabellen

Beschreiben die Fakten (Wer? Was? Wo? Wann?). Wenige Zeilen, viele Spalten. Beispiele: Datum, Produkt, Kunde, Region, Mitarbeiter.

💡
Goldene Regel: Filter fließen von Dimensionstabellen → in Faktentabellen. Niemals Faktentabellen direkt miteinander verbinden (viele-zu-viele vermeiden).
DAX – Many-to-Many via Bridge-Tabelle lösen 
-- Problem: Produkte können mehrere Kategorien haben (m:n)
-- Lösung: Bridge-Tabelle (Produkt_Kategorie)

-- Modell: Kategorie (1) → (n) Bridge (n) → (1) Produkt
-- Alle Beziehungen: 1:n, beide Richtungen single

-- DAX: Measure mit CROSSFILTER für bidirektionale Filterung
Umsatz nach Kategorie =
CALCULATE(
    [Umsatz],
    CROSSFILTER(Produkt_Kategorie[ProduktID], Verkauf[ProduktID], Both)
)
Modellierungs-Checkliste
  • Keine berechneten Spalten in Faktentabellen (nutze stattdessen Power Query)
  • Datumstabelle immer als berechnete Tabelle (keine aus Faktentabelle abgeleitete)
  • Primärschlüssel in Dimensionen sind immer Integer (schneller als Text)
  • Niemals Spalten importieren die nicht gebraucht werden (Modellgröße!)
  • Duplikate in Dimensionstabellen entfernen (Power Query: Remove Duplicates)
  • Beziehungsrichtung: Standard = Single (von Dim → Fakt)
  • Bidirektionale Beziehungen nur wenn zwingend nötig (Performance-Impact)

Power Query – Fortgeschrittener M-Code

Die grafische Power Query GUI erzeugt M-Code im Hintergrund. Direktes Schreiben von M-Code ermöglicht Transformationen die in der GUI nicht möglich sind.

M-Code – Dynamische Parameterabfrage 
// Parameter als Query definieren (Name: StartJahr, Typ: Number)
// Dann in anderen Queries referenzieren:

let
    Quelle = Excel.CurrentWorkbook(){[Name="Verkaufsdaten"]}[Content],

    // Parametergesteuerte Filterung
    Gefiltert = Table.SelectRows(Quelle,
        each Date.Year([Datum]) >= StartJahr),

    // Dynamisch Spalten pivotieren
    Pivotiert = Table.Pivot(
        Gefiltert,
        List.Distinct(Gefiltert[Kategorie]),
        "Kategorie",
        "Umsatz",
        List.Sum
    ),

    // Typ dynamisch setzen (alle numerischen Spalten)
    NumSpalten = List.Select(
        Table.ColumnNames(Pivotiert),
        each List.Contains(
            List.Distinct(Gefiltert[Kategorie]), _)
    ),
    TypTransform = Table.TransformColumnTypes(
        Pivotiert,
        List.Transform(NumSpalten, each {_, type number})
    )
in
    TypTransform
M-Code – Ordner-Konsolidierung (alle CSVs laden) 
let
    Ordner = Folder.Files("C:\Monatsdaten\"),

    // Nur CSV-Dateien
    NurCSV = Table.SelectRows(Ordner,
        each Text.EndsWith([Name], ".csv")),

    // Funktion: Eine CSV-Datei laden und transformieren
    LadeEineCSV = (dateiinhalt) =>
        let
            geladen  = Csv.Document(dateiinhalt, [Delimiter=";", Encoding=65001]),
            benamst  = Table.PromoteHeaders(geladen),
            getypt   = Table.TransformColumnTypes(benamst,{
                           {"Datum", type date},
                           {"Betrag", type number}})
        in getypt,

    // Auf alle Dateien anwenden und kombinieren
    Transformiert = Table.AddColumn(NurCSV, "Daten",
        each LadeEineCSV([Content])),
    Kombiniert    = Table.Combine(Transformiert[Daten])
in
    Kombiniert

DAX – Conditional Formatting (Farbe per Measure) 
-- Measure für Hintergrundfarbe (als Hex-String)
KPI Farbe =
VAR Wachstum = [Wachstum % VJ]
RETURN
SWITCH(
    TRUE(),
    Wachstum >= 0.15,  "#107C10",  -- Dunkelgrün
    Wachstum >= 0.05,  "#4EC94E",  -- Hellgrün

Row-Level Security (RLS)

RLS beschränkt den Datenzugriff auf Zeilenebene. Benutzer sehen im selben Report nur die Daten die für ihre Rolle freigegeben sind.

🔒

Statische RLS

Feste DAX-Filterregel je Rolle. Benutzer werden der Rolle im Service zugewiesen. Einfach, aber unflexibel bei vielen Regionen/Abteilungen.

👤

Dynamische RLS

DAX-Regel nutzt USERNAME() oder USERPRINCIPALNAME() und vergleicht mit einer Benutzer-Tabelle im Modell. Eine Rolle für alle – skaliert beliebig.

DAX – Statische RLS (Modeling → Manage Roles) 
-- Rolle: "Region_Nord"
-- DAX-Filterregel auf Tabelle "Region":
[Region] = "Nord"

-- Rolle: "Eigene_Daten" (statisch nach Abteilung)
[Abteilung] = "Vertrieb"
DAX – Dynamische RLS (skaliert auf beliebig viele User) 
-- Benutzer-Tabelle im Modell: [Email] | [Region] | [Darf_Alles]
-- Verknüpfung: Benutzer[Region] → Region[Region] (inaktiv oder via TREATAS)

-- Rolle: "Dynamisch" – DAX-Filter auf Tabelle "Region":
[Region] =
LOOKUPVALUE(
    Benutzer[Region],          -- Rückgabewert
    Benutzer[Email],           -- Suchspalte
    USERPRINCIPALNAME()        -- Aktueller eingeloggter User (Azure AD Email)
)
-- || ... ODER Admin-Ausnahme:
|| LOOKUPVALUE(Benutzer[Darf_Alles], Benutzer[Email], USERPRINCIPALNAME()) = TRUE()

-- Testen: Modeling → View as → wähle Rolle → tritt als User auf
-- Im Service: Dataset → Security → Rolle → User hinzufügen
✅
RLS im Service testen: Workspace → Dataset → Sicherheit → Rollen testen (als Benutzer anmelden). Im Desktop: Modeling → View as Roles.

☁️ Power BI Service – Fortgeschrittene Cloud-Features

Der Power BI Service bietet weit mehr als nur das Veröffentlichen von Reports. Diese Features sind nur in der Cloud verfügbar.

🔄

Deployment Pipelines

Dev → Test → Prod Workflow für professionelle BI-Entwicklung. Änderungen werden in Dev entwickelt, in Test getestet und in Prod deployed – ohne manuelle Veröffentlichung.

🌊

Dataflows

Zentralisierte ETL-Schicht in der Cloud. Power Query in der Cloud ausführen, Ergebnis als wiederverwendbare Datentabelle bereitstellen – mehrere Datasets nutzen denselben Dataflow.

📄

Paginated Reports

Pixel-perfekte, seitenbasierte Reports (wie SSRS). Für Druckberichte, Rechnungen, mehrseitige Exporte. Erstellt mit Power BI Report Builder.

🤖

AI Features

Q&A (natürliche Sprache), Smart Narrative (automatische Textbeschreibungen), Key Influencers Visual, Anomaly Detection – alle Cloud-only.

  • 1

    Pipeline erstellen

    Power BI Service → Deployment Pipelines (linke Sidebar) → Pipeline erstellen → Name vergeben.

  • 2

    Workspaces zuweisen

    Drei Stufen: Development, Test, Production → je einen dedizierten Workspace zuweisen (Premium-Kapazität erforderlich).

    💡 Workspace-Namen Konvention: [Projektname] - Dev, [Projektname] - Test, [Projektname] - Prod

    Deployment Rules konfigurieren

    Datenquellen pro Stage unterschiedlich konfigurieren (Dev-Datenbank vs. Prod-Datenbank) ohne den Report zu ändern. Pipeline → Rules → Datasource Rule → pro Stage andere Connection String.

⚡
Gateway für lokale Datenquellen: Wenn Daten aus einer lokalen SQL-Datenbank, einem Netzlaufwerk oder einer lokalen Excel-Datei kommen, muss ein On-Premises Data Gateway installiert werden. Download: aka.ms/datagateway → Im Service: Einstellungen → Gateways verwalten → Dataset → Gateway verbinden.
