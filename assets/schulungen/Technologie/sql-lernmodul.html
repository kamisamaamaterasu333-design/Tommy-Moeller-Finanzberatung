SQL & Datenbankgrundlagen

Von der ersten SELECT-Abfrage bis zu komplexen JOINs und Aggregationen – SQL als Kernwerkzeug für Controller und Datenanalysten.

Was sind Datenbanken?

Grundkonzepte, Datenbanktypen und warum SQL für Analysten unverzichtbar ist.

Relationale Datenbanken – das Fundament

Eine relationale Datenbank speichert Daten in Tabellen (Relationen) – ähnlich wie Excel-Tabellenblätter, aber mit strikter Struktur, Beziehungen zwischen Tabellen und optimierter Abfrageperformance für Millionen von Datensätzen.

💡 SQL (Structured Query Language) ist die Sprache, um mit relationalen Datenbanken zu kommunizieren. Einmal gelernt, läuft SQL auf allen gängigen Systemen: PostgreSQL, MySQL, SQLite, SQL Server, Snowflake, BigQuery.

Tabelle (Relation)

Daten in Zeilen (Records) und Spalten (Felder). Jede Tabelle beschreibt eine Entität: Kunden, Verträge, Transaktionen.

Primärschlüssel (PK)

Eindeutige ID pro Zeile – kein Duplikat möglich. Meist eine auto-incrementierte Zahl oder UUID.

Fremdschlüssel (FK)

Verknüpft zwei Tabellen. Ein Kreditvertrag hat eine Kunden-ID, die auf die Kundentabelle zeigt.

Index

Wie ein Register im Buch – macht Abfragen auf großen Tabellen drastisch schneller.

Datenbanktypen im Überblick

Relational (SQL)

PostgreSQL, MySQL, SQL Server

  • Strukturierte Daten mit festen Schemas
  • Starke Konsistenzgarantien (ACID)
  • Standard in Banken, ERP, Finanzsystemen
  • Ideal für transaktionale Daten
NoSQL / Document

MongoDB, Supabase (PostgreSQL+)

  • Flexible Datenstrukturen (JSON)
  • Gut für unstrukturierte / variable Daten
  • Schnell bei einfachen Lesevorgängen
  • Supabase kombiniert SQL + Flexibilität

SELECT – Daten abfragen

Die wichtigsten SQL-Befehle: von einfachen Abfragen bis zu gefilterten, sortierten Resultaten.

Die SELECT-Grundstruktur

Jede SQL-Abfrage folgt dieser logischen Reihenfolge:

SQL
SELECT -- Was willst du sehen? (Spalten) FROM -- Woher? (Tabelle) WHERE -- Welche Zeilen? (Filter) GROUP BY -- Gruppieren nach HAVING -- Filter auf Gruppen ORDER BY -- Sortierung LIMIT -- Maximale Anzahl Zeilen
⚠️ Die Schreibreihenfolge (oben) ≠ Ausführungsreihenfolge! SQL verarbeitet intern: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT.

Erste Abfragen

Alle Daten einer Tabelle

SQL
SELECT * FROM kunden; -- Nur bestimmte Spalten: SELECT kunden_id, name, rating FROM kunden;

Filtern mit WHERE

SQL
-- Verträge über 500.000 € SELECT vertrag_id, volumen, zinssatz FROM vertraege WHERE volumen > 500000; -- Mehrere Bedingungen (AND / OR) SELECT * FROM vertraege WHERE volumen > 500000 AND status = 'aktiv'; -- IN-Liste: mehrere Werte prüfen SELECT * FROM kunden WHERE rating IN ('A', 'AA', 'AAA'); -- BETWEEN: Wertebereich SELECT * FROM zahlungen WHERE datum BETWEEN '2024-01-01' AND '2024-12-31'; -- LIKE: Textsuche (% = beliebige Zeichen) SELECT * FROM kunden WHERE name LIKE 'Müller%';

Sortieren und begrenzen

SQL
-- Top 10 größte Verträge SELECT vertrag_id, volumen, zinssatz FROM vertraege ORDER BY volumen DESC LIMIT 10;

Nützliche Funktionen

SQL
-- Textfunktionen SELECT UPPER(name) AS name_gross, LOWER(name) AS name_klein, LENGTH(name) AS zeichenanzahl, TRIM(name) AS ohne_leerzeichen FROM kunden; -- Datumsfunktionen (PostgreSQL / Supabase) SELECT datum, EXTRACT(YEAR FROM datum) AS jahr, EXTRACT(MONTH FROM datum) AS monat, DATE_TRUNC('month', datum) AS monatsbeginn FROM zahlungen; -- CASE WHEN: bedingte Logik (wie IF in Excel) SELECT vertrag_id, volumen, CASE WHEN volumen >= 1000000 THEN 'Großkredit' WHEN volumen >= 100000 THEN 'Mittelgroß' ELSE 'Kleinkredit' END AS groessenklasse FROM vertraege;

JOINs – Tabellen verknüpfen

Der mächtigste Aspekt von SQL: Daten aus mehreren Tabellen sinnvoll zusammenführen.

JOIN-Typen erklärt

INNER JOIN

Nur Zeilen, die in beiden Tabellen übereinstimmen. Nicht gematchte Zeilen werden weggelassen.

LEFT JOIN

Alle Zeilen der linken Tabelle + passende aus der rechten. Kein Match → NULL-Werte rechts.

RIGHT JOIN

Wie LEFT JOIN, aber umgekehrt. Selten verwendet – meist schreibt man stattdessen einen LEFT JOIN.

FULL OUTER JOIN

Alle Zeilen beider Tabellen, unabhängig ob ein Match existiert. NULLs wo kein Partner.

💡 Für 90% aller Analysen brauchst du nur INNER JOIN und LEFT JOIN. LEFT JOIN ist besonders nützlich, wenn du prüfen willst, ob Datensätze in einer anderen Tabelle fehlen.

JOINs in der Praxis

INNER JOIN – Kunden mit Verträgen

SQL
-- Nur Kunden MIT Vertrag erscheinen SELECT k.name, k.rating, v.vertrag_id, v.volumen, v.zinssatz FROM kunden k INNER JOIN vertraege v ON k.kunden_id = v.kunden_id WHERE v.status = 'aktiv' ORDER BY v.volumen DESC;

LEFT JOIN – Kunden ohne Vertrag finden

SQL
-- Alle Kunden, auch ohne Vertrag (NULL = kein Vertrag) SELECT k.name, k.segment, v.vertrag_id, -- NULL wenn kein Vertrag v.volumen FROM kunden k LEFT JOIN vertraege v ON k.kunden_id = v.kunden_id; -- Nur Kunden OHNE Vertrag: SELECT k.name, k.segment FROM kunden k LEFT JOIN vertraege v ON k.kunden_id = v.kunden_id WHERE v.vertrag_id IS NULL;

3-Tabellen-JOIN – vollständiger Zahlungsbericht

SQL
SELECT k.name AS kunde, v.vertrag_id, v.volumen, z.datum, z.betrag AS zahlung_betrag, z.typ AS zahlungstyp FROM kunden k INNER JOIN vertraege v ON k.kunden_id = v.kunden_id INNER JOIN zahlungen z ON v.vertrag_id = z.vertrag_id WHERE z.datum >= '2024-01-01' ORDER BY k.name, z.datum;

Aggregation & Analyse

GROUP BY, Fensterfunktionen und CTEs – die Kernwerkzeuge für Reporting und Controlling.

GROUP BY – Daten zusammenfassen

SQL
-- Aggregationsfunktionen SELECT segment, COUNT(*) AS anzahl_kunden, COUNT(v.vertrag_id) AS anzahl_vertraege, SUM(v.volumen) AS gesamtvolumen, AVG(v.zinssatz) AS durchschnittszins, MIN(v.volumen) AS kleinstes_volumen, MAX(v.volumen) AS groesstes_volumen FROM kunden k LEFT JOIN vertraege v ON k.kunden_id = v.kunden_id GROUP BY segment HAVING SUM(v.volumen) > 1000000 -- Filter auf Gruppen! ORDER BY gesamtvolumen DESC;
⚠️ WHERE vs. HAVING: WHERE filtert einzelne Zeilen vor der Gruppierung. HAVING filtert Gruppen nach der Aggregation. Du kannst nicht WHERE SUM(...) > ... schreiben – das ist ein häufiger Fehler.

Fensterfunktionen (Window Functions)

Fensterfunktionen berechnen Werte über eine Gruppe von Zeilen, ohne diese zusammenzufassen – wie eine laufende Summe in Excel.

SQL
-- Rang nach Volumen pro Segment SELECT k.name, k.segment, v.volumen, RANK() OVER ( PARTITION BY k.segment ORDER BY v.volumen DESC ) AS rang_im_segment FROM kunden k JOIN vertraege v ON k.kunden_id = v.kunden_id; -- Laufende Summe der Zahlungen (YTD) SELECT datum, betrag, SUM(betrag) OVER ( ORDER BY datum ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW ) AS kumulierte_zahlungen FROM zahlungen WHERE EXTRACT(YEAR FROM datum) = 2024;

CTEs – lesbare komplexe Abfragen

Common Table Expressions (WITH-Klauseln) machen lange Abfragen übersichtlich. Wie benannte Zwischentabellen.

SQL
-- Portfolio-Risikobericht mit CTE WITH portfolio_summary AS ( SELECT k.segment, k.rating, COUNT(*) AS anzahl, SUM(v.volumen) AS volumen_gesamt FROM kunden k JOIN vertraege v ON k.kunden_id = v.kunden_id WHERE v.status = 'aktiv' GROUP BY k.segment, k.rating ), gesamtvolumen AS ( SELECT SUM(volumen_gesamt) AS total FROM portfolio_summary ) SELECT ps.segment, ps.rating, ps.anzahl, ps.volumen_gesamt, ROUND(ps.volumen_gesamt * 100.0 / gv.total, 2) AS anteil_pct FROM portfolio_summary ps CROSS JOIN gesamtvolumen gv ORDER BY ps.volumen_gesamt DESC;

Praxisfälle für Controller

Typische SQL-Aufgaben im Controlling und Finanzwesen – direkt anwendbar.

Monatliches Reporting

SQL
-- Monatliche Zahlungseingänge nach Jahr/Monat SELECT EXTRACT(YEAR FROM datum) AS jahr, EXTRACT(MONTH FROM datum) AS monat, COUNT(*) AS anzahl_zahlungen, SUM(betrag) AS summe_eingang, AVG(betrag) AS durchschnitt FROM zahlungen WHERE typ = 'Tilgung' GROUP BY jahr, monat ORDER BY jahr, monat;

Vorjahresvergleich

SQL
WITH monatsdaten AS ( SELECT DATE_TRUNC('month', datum) AS monat, SUM(betrag) AS volumen FROM zahlungen GROUP BY 1 ) SELECT akt.monat, akt.volumen AS aktuell, vor.volumen AS vorjahr, akt.volumen - vor.volumen AS differenz, ROUND((akt.volumen - vor.volumen) * 100.0 / vor.volumen, 1) AS veraenderung_pct FROM monatsdaten akt LEFT JOIN monatsdaten vor ON akt.monat = vor.monat + INTERVAL '1 year' ORDER BY akt.monat DESC;

SQL in Power BI

Du kannst SQL-Abfragen direkt als Power BI-Datenquelle nutzen (DirectQuery oder Import). Das ersetzt komplexe Power Query-Schritte und ist oft schneller.

  • Aggregationen in SQL statt in Power BI = schnellere Reports
  • JOINs in SQL statt Beziehungen in Power BI = einfacheres Datenmodell
  • CTEs als Grundlage für Measures = sauberere Architektur

Wissenscheck SQL

7 Fragen – teste dein SQL-Verständnis.

1. Was ist der Unterschied zwischen WHERE und HAVING?
WHERE filtert einzelne Datensätze vor der Gruppierung. HAVING filtert Gruppen nach der Aggregation – du kannst z.B. HAVING SUM(volumen) > 1000000 schreiben, aber nicht WHERE SUM(volumen) > 1000000.
2. Du willst alle Kunden sehen, auch solche OHNE Vertrag. Welcher JOIN ist richtig?
LEFT JOIN gibt alle Zeilen der linken Tabelle zurück (hier: Kunden), auch wenn kein passender Eintrag rechts (Verträge) existiert. Kunden ohne Vertrag erscheinen mit NULL-Werten in den Vertragsspalten.
3. Was macht DATE_TRUNC('month', datum)?
DATE_TRUNC('month', datum) gibt den Monatsbeginn zurück. Damit kannst du Daten nach Monat gruppieren, ohne EXTRACT verwenden zu müssen – sehr praktisch für monatliche Berichte.
4. Was ist ein CTE (Common Table Expression)?
CTEs (WITH klausel_name AS (...)) sind temporäre, benannte Abfrageergebnisse, die nur für die Dauer der übergeordneten Abfrage existieren. Sie machen komplexe Abfragen lesbar und wartbar.
5. Wofür steht das * in SELECT * FROM kunden?
SELECT * bedeutet "wähle alle Spalten". In der Produktion sollte man konkrete Spaltennamen angeben statt * – das verbessert Performance und Wartbarkeit.
6. Du willst die Top-3-Kunden nach Kreditvolumen je Segment. Welche Funktion hilft?
Fensterfunktionen wie RANK() OVER (PARTITION BY ...) erstellen einen Rang innerhalb jeder Gruppe (Partition), ohne die Zeilen zusammenzufassen. Danach filterst du WHERE rang <= 3.
7. Was ist der Primärschlüssel (Primary Key)?
Der Primärschlüssel (PK) identifiziert jede Zeile eindeutig. Andere Tabellen verweisen über Fremdschlüssel (FK) darauf. Typisch: id SERIAL PRIMARY KEY in PostgreSQL.
