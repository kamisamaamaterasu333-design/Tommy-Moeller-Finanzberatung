Was ist eine REST API?

Das Konzept hinter APIs – und warum du als Controller und Entwickler davon wissen musst.

APIs – die Sprache der modernen Softwarewelt

Eine API (Application Programming Interface) ist eine definierte Schnittstelle, über die zwei Systeme miteinander kommunizieren. REST (Representational State Transfer) ist der heute dominante Architekturstil für Web-APIs.

🍽️ Die beste Analogie: Du sitzt im Restaurant. Die Speisekarte ist die API-Dokumentation. Du (Client) gibst dem Kellner (API) deine Bestellung. Er bringt sie in die Küche (Server) und liefert das Ergebnis. Du musst nicht wissen, wie die Küche arbeitet.

REST-Prinzipien

Ressourcen (URLs)

Alles ist eine Ressource

  • /kunden – alle Kunden
  • /kunden/42 – Kunde mit ID 42
  • /kunden/42/vertraege – Verträge von Kunde 42
  • Substantive, nicht Verben in URLs
Zustandslos (Stateless)

Jeder Request ist unabhängig

  • Server speichert keinen Session-State
  • Jeder Request enthält alle nötigen Infos
  • Einfacher zu skalieren und zu cachen
  • Auth-Token muss jedes Mal mitgesendet werden
🔗 Deine Supabase-Datenbank auf tommy-moeller.de hat eine vollständige REST API – automatisch generiert für jede Tabelle. URL-Muster: https://[projekt].supabase.co/rest/v1/[tabellenname]

HTTP-Methoden & JSON

Die Sprache von REST: Was GET, POST, PUT, DELETE bedeuten – und wie JSON-Daten aussehen.

HTTP-Methoden (Verben)

GET

Daten abrufen. Keine Änderung am Server. Sicher und wiederholbar.

POST

Neuen Datensatz erstellen. Body enthält die neuen Daten als JSON.

PUT

Datensatz vollständig ersetzen. Alle Felder müssen mitgegeben werden.

PATCH

Datensatz teilweise aktualisieren. Nur geänderte Felder nötig.

DELETE

Datensatz löschen. Vorsicht: oft unwiderruflich!

💡 Die Eselsbrücke: CRUD (Create/Read/Update/Delete) entspricht POST / GET / PUT+PATCH / DELETE. Das sind die vier Grundoperationen jeder Datenverwaltung.

HTTP Status-Codes

Der Server antwortet immer mit einem 3-stelligen Status-Code. Die erste Ziffer zeigt die Kategorie:

200OK – Anfrage erfolgreich
201Created – Ressource erstellt
204No Content – Erfolgreich, kein Rückgabewert
301Moved Permanently – Weitergeleitet
400Bad Request – Fehlerhafte Anfrage
401Unauthorized – Nicht authentifiziert
403Forbidden – Keine Berechtigung
404Not Found – Ressource nicht gefunden
429Too Many Requests – Rate Limit erreicht
500Internal Server Error – Serverfehler
503Service Unavailable – Server überlastet

JSON – das Datenformat von REST

JSON (JavaScript Object Notation) ist das universelle Datenaustauschformat. Leicht lesbar für Menschen und Maschinen.

JSON – Beispiel Kreditportfolio
{ "kunden_id": 42, "name": "Müller GmbH", "segment": "KMU", "rating": "B+", "aktiv": true, "vertraege": [ { "vertrag_id": 101, "volumen": 250000.00, "zinssatz": 4.8, "status": "aktiv" }, { "vertrag_id": 102, "volumen": 75000.00, "zinssatz": 5.1, "status": "beendet" } ], "adresse": { "strasse": "Musterstr. 12", "plz": "10115", "stadt": "Berlin" }, "notizen": null }

JSON-Datentypen

Einfache Typen

  • "text" → String (in Anführungszeichen)
  • 42 oder 4.8 → Number
  • true / false → Boolean
  • null → kein Wert / leer

Strukturierte Typen

  • {} → Object (Schlüssel-Wert-Paare)
  • [] → Array (Liste von Werten)
  • Arrays können Objects enthalten
  • Beliebig tief verschachtelbar

API-Authentifizierung

Wie APIs prüfen, wer du bist – und warum Sicherheit dabei so wichtig ist.

Die wichtigsten Auth-Methoden

API-Key

Einfachste Methode

  • Ein geheimer Schlüssel im Header
  • Kein Login nötig, immer gültig
  • Gut für Server-zu-Server
  • Risiko: Key geleakt = alles offen
  • Beispiel: Supabase, Anthropic, Google
Bearer Token (JWT)

Standard für User-Auth

  • Zeitlich begrenzt (z.B. 1 Stunde)
  • Enthält User-Infos (Name, Rolle)
  • Signiert – kann nicht gefälscht werden
  • Standard bei Supabase Auth
  • Refresh Token verlängert Session
OAuth 2.0

"Login mit Google/GitHub"

  • Nutzer autorisiert deine App
  • Kein Passwort an dich weitergegeben
  • Standard für Drittanbieter-Login
  • Supabase Auth unterstützt OAuth
Basic Auth

Username + Passwort

  • Base64-kodiert im Header
  • Nur über HTTPS sicher!
  • Veraltet – möglichst vermeiden
  • Noch in alten Systemen vorhanden
🔒 Goldene Regel: API-Keys und Tokens niemals im Frontend-Code oder in Git-Repositories speichern. Nutze Umgebungsvariablen (.env-Dateien) oder Secrets-Manager. Dein Supabase anon-key im HTML ist eine Ausnahme – er ist für öffentlichen Zugriff konzipiert und durch Row Level Security abgesichert.

Webhooks – Push statt Pull

Statt regelmäßig zu fragen "Gibt es Neues?" – einfach benachrichtigt werden wenn etwas passiert.

API (Pull) vs. Webhook (Push)

Klassische API (Pull)

Du fragst aktiv nach

  • Dein Code fragt alle X Minuten an
  • Meist unnötige Anfragen (nichts Neues)
  • Gut wenn du Kontrolle über Timing brauchst
  • Gut für einmalige Abfragen
Webhook (Push)

Du wirst benachrichtigt

  • Service sendet HTTP-Request an dich
  • Nur wenn wirklich etwas passiert
  • Viel effizienter (keine Polling-Loops)
  • Gut für Events: Zahlung, Formulareingabe
📞 Analogie: API-Pull ist wie alle 5 Minuten die Postfach zu checken. Webhook ist wie eine Klingel an der Haustür – die Post klingelt, wenn das Paket da ist.

Supabase Edge Function – Webhook-Handler
// supabase/functions/webhook-handler/index.ts import { serve } from "https://deno.land/std/http/server.ts"; serve(async (req: Request) => { // Nur POST-Requests erlauben if (req.method !== "POST") { return new Response("Method Not Allowed", { status: 405 }); } // Signature prüfen (Sicherheit!) const signature = req.headers.get("x-webhook-signature"); if (!isValidSignature(signature)) { return new Response("Unauthorized", { status: 401 }); } // Body parsen const event = await req.json(); // Event verarbeiten if (event.type === "zahlung.eingegangen") { await aktualisierePortfolio(event.data); await sendeBestaetigung(event.data.email); } return new Response("OK", { status: 200 }); });
🔒 Webhook-Sicherheit: Immer die Signatur des Senders prüfen! Sonst könnte jeder beliebige POST-Requests an deinen Endpoint schicken. Dienste wie Stripe, GitHub und Supabase signieren Webhooks mit HMAC-SHA256.

Webhook-Anwendungsfälle

  • Formular auf tommy-moeller.de ausgefüllt → automatische Bestätigungs-E-Mail + Eintrag in Datenbank
  • GitHub-Commit in main → automatischer Deploy auf GitHub Pages
  • Stripe-Zahlung bestätigt → Freischaltung von Lernmodulen in Supabase
  • Supabase-Datenbankänderung → Benachrichtigung via E-Mail oder Slack
  • Power Automate: neuer SharePoint-Eintrag → Teams-Benachrichtigung + Excel-Update

Wissenscheck REST APIs

8 Fragen zu HTTP, JSON, Auth und Webhooks.

1. Was bedeutet HTTP-Statuscode 404?
404 Not Found bedeutet, dass die URL zwar erreichbar ist, aber die spezifische Ressource nicht existiert – z.B. /kunden/999 wenn Kunde 999 nicht in der Datenbank ist.
2. Welche HTTP-Methode ist für das Erstellen eines neuen Datensatzes gedacht?
POST erstellt neue Ressourcen. GET liest, PUT ersetzt vollständig, PATCH aktualisiert teilweise, DELETE löscht. Das CRUD-Modell: Create=POST, Read=GET, Update=PUT/PATCH, Delete=DELETE.
3. Was ist der Hauptunterschied zwischen PUT und PATCH?
PUT: Du sendest den kompletten Datensatz – alle Felder. PATCH: Du sendest nur die zu ändernden Felder. Für partielles Update (z.B. nur Status ändern) ist PATCH effizienter und sicherer.
4. Was ist der Hauptvorteil von Webhooks gegenüber regelmäßigem API-Polling?
Polling (regelmäßiges Abfragen) verbraucht Ressourcen auch wenn nichts passiert. Webhooks (Push) reagieren nur auf echte Events – effizienter und reaktiver.
5. Warum sollte ein API-Key für Gemini/Anthropic nie im Frontend-Code stehen?
JavaScript-Code im Browser ist für jeden einsehbar (F12 → Quelltext). Ein eingebetteter API-Key kann sofort missbraucht werden. Lösung: Key serverseitig in Edge Functions / Backend halten.
6. Was ist JSON?
JSON (JavaScript Object Notation) ist das universelle Format für Datenübertragung im Web. Es besteht aus Objekten {} mit Schlüssel-Wert-Paaren und Arrays []. Lesbar für Menschen und Maschinen.
7. Was passiert bei einem 401-Fehler?
401 Unauthorized bedeutet: kein gültiger Auth-Token. 403 Forbidden bedeutet: Token ist gültig, aber du hast keine Berechtigung. 429 Too Many Requests ist das Rate-Limit.
8. Was ist der Zweck einer Webhook-Signatur?
Ohne Signaturprüfung könnte jeder an deine Webhook-URL POST-Requests schicken und so Events fälschen. HMAC-SHA256-Signaturen stellen sicher, dass der Sender den geheimen Key kennt.
