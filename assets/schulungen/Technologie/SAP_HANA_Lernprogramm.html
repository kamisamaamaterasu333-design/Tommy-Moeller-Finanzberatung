Kapitel 1 · Grundlagen

SAP HANA Architektur

SAP HANA (High-Performance Analytic Appliance) ist eine In-Memory-Datenbank- und Applikationsplattform. Sie wurde 2010 eingeführt und vereint OLTP und OLAP in einem System.

Kernkomponenten im Überblick

🧠
Index Server
Hauptkomponente: verwaltet Datenspeicher, SQL-Verarbeitung und Transaktionsmanagement. Enthält Row Store, Column Store und In-Memory Engine.
📡
XS Engine (Extended Application Services)
Eingebetteter Web-Server direkt in HANA. Erlaubt Entwicklung von Web-Apps direkt auf der Datenbank. Heute als XSA (Advanced) Architektur.
🔄
Preprocessor Server
Verarbeitet unstrukturierte Daten und Text für die HANA Search. Ermöglicht Volltext-Suche und linguistische Analyse.
📊
Statistics Server
Sammelt Systemmetriken und Performance-Daten. Grundlage für Monitoring und automatische Optimierung.
🔐
Name Server
Verwaltet die Topologie von HANA-Systemen. In verteilten Umgebungen koordiniert er, welcher Knoten welche Daten hält.
📁
Compile Server
Übersetzt HANA-Skripte (HEX, L-Code) in ausführbaren Maschinencode für maximale Laufzeitperformance.
Schlüsselkonzept: HANA ist kein klassisches RDBMS. Die Architektur ist so designed, dass Analysen und Transaktionen gleichzeitig auf denselben Daten laufen – ohne Datenkopien oder ETL-Prozesse zwischen OLTP und OLAP.

🎯 Quick-Check

Was ist der Hauptunterschied zwischen dem Index Server und dem XS Engine?

✓ Richtig! Der Index Server ist das Herzstück der Datenhaltung, während die XS Engine Web-Applikationen direkt auf der Datenbank ermöglicht.
✗ Nicht ganz. Der Index Server ist der Core für Daten, SQL und Transaktionen. Die XS Engine ist ein eingebetteter Webserver für Applikationen direkt auf HANA.
Kapitel 2 · Grundlagen

Das In-Memory Konzept

Das zentrale Paradigma von SAP HANA: alle relevanten Daten liegen im RAM statt auf der Festplatte. Das verändert alles – Architektur, Performance und Möglichkeiten.

Column Store vs. Row Store

Kriterium Column Store (HANA Standard) Row Store (traditionell)
DatenspeicherungSpaltenweise (alle Werte einer Spalte zusammen)Zeilenweise (ein Datensatz zusammenhängend)
Analytische Abfragen⚡ Extrem schnell (nur relevante Spalten lesen)🐢 Langsam (gesamte Zeilen durchsuchen)
Transaktionen (INSERT/UPDATE)Gut (mit Delta-Store-Mechanismus)⚡ Schnell (direkte Zeilen-Manipulation)
KompressionHoch (gleiche Werte in Spalten komprimierbar)Gering (heterogene Zeileninhalte)
Typischer EinsatzOLAP, Analytics, ReportsOLTP, einfache Lookups

Der Delta-Store: Brücke zwischen OLTP und OLAP

TECHNISCHES KONZEPT

HANA verwendet einen zweistufigen Speicher: Neue Daten landen zuerst im Delta Store (zeilenoptimiert für schnelle Schreibvorgänge). Periodisch wird ein Delta Merge durchgeführt, der diese Daten in den komprimierten Main Store (spaltenoptimiert) überführt. Abfragen werden automatisch über beide Schichten aggregiert – für den Nutzer transparent.

Datenkompression in HANA

Dictionary Encoding
Wiederkehrende Werte werden durch kurze Integer-Codes ersetzt. "Deutschland" → 1. Spart massiv Speicher bei Stammdaten.
Run-Length Encoding
Aufeinanderfolgende gleiche Werte werden als (Wert, Anzahl) kodiert. "AAA" → (A,3). Ideal für sortierte Daten.
Prefix Encoding
Häufige Präfixe werden ausgelagert. Spart Platz bei Textfeldern mit ähnlichem Anfang wie Produktcodes.

🎯 Quick-Check

Warum ist Column Store für analytische Abfragen schneller als Row Store?

✓ Genau! Bei "SELECT SUM(Umsatz)" muss Column Store nur die Umsatz-Spalte lesen – Row Store muss alle Zeilen mit allen Feldern scannen.
✗ RAM allein ist nicht der Faktor. Der entscheidende Vorteil liegt im Zugriffsmuster: Spaltenweise Speicherung bedeutet, dass Aggregationen nur die relevanten Datenbereiche lesen.
Kapitel 3 · Grundlagen

Datenmodellierung in HANA

HANA ersetzt traditionelle ABAP-Aggregat-Views durch virtuelle Informationsmodelle. Das verändert grundlegend, wie Geschäftsdaten strukturiert und abgefragt werden.

HANA Views – die Informationsmodelle

📋
Attribute View
(Veraltet in HANA 2.0) Beschreibende Stammdaten-Dimensionen. Wurde durch CDS Views abgelöst. Vergleichbar mit Master-Data-Tabellen in BW.
📈
Analytic View
(Veraltet) Faktentabellen mit Kennzahlen. Kombinierte Messwerte mit Dimensionen. Heute: Calculation Views übernehmen diese Rolle.
🔢
Calculation View
Das moderne Herzstück der HANA-Modellierung. Unterstützt komplexe Berechnungen, Joins, Unions, Aggregationen – graphisch oder in SQLScript.
🗂️
CDS Views (Core Data Services)
Deklarative Datendefinitionen in ABAP oder HANA. Standard für S/4HANA. Definieren Datenzugriff und -semantik auf Applikationsebene.

SQLScript – Programmierung auf der Datenbank

-- SQLScript: Berechnung direkt auf HANA-Ebene CREATE PROCEDURE GetTopCustomers ( IN p_year INTEGER, OUT customers TABLE(CustomerID NVARCHAR(10), Revenue DECIMAL(15,2)) ) LANGUAGE SQLSCRIPT AS BEGIN customers = SELECT c.CustomerID, SUM(o.NetAmount) AS Revenue FROM Orders o JOIN Customers c ON o.CustomerID = c.CustomerID WHERE YEAR(o.OrderDate) = :p_year GROUP BY c.CustomerID ORDER BY Revenue DESC LIMIT 10; END;
Code Pushdown Prinzip: In HANA wird Logik so weit wie möglich auf Datenbankebene ausgeführt ("pushed down"), statt Daten in die Applikationsschicht zu laden. Das reduziert Netzwerk-Traffic und nutzt HANAs Parallelverarbeitung optimal.

S/4HANA ist der direkte Nachfolger von SAP ECC (auf Basis von R/3). Es ist ausschließlich auf HANA lauffähig und wurde von Grund auf neu designed.

S/4HANA Finance (Central Finance)

📒
Universal Journal (ACDOCA)
Eine einzige Tabelle für alle buchhalterischen Sichten: FI, CO, ML, AA, PS. Keine getrennten Ledger mehr – ein einheitliches Datenmodell.
⚡
Echtzeit-Abschluss
Monatsabschlüsse dauern Stunden statt Tage. Period-End-Closing kann durch kontinuierliche Buchführung vereinfacht werden.
🔄
Integrated Business Planning
Finance und Controlling auf einer Datenquelle. Budgetplanung, Forecast und Actuals auf denselben Strukturen.

S/4HANA Logistik & Supply Chain

📦
Material Management (MM)
Vereinfachtes Datenmodell: MATDOC statt MSEG/MKPF. Echtzeit-Lagerbestand ohne Aggregat-Tabellen.

Production Planning (PP)

🚚
Sales & Distribution (SD)
Available-to-Promise (ATP) in Echtzeit. Lieferterminbestätigung auf Basis tatsächlicher Lagerbestände, nicht aggregierter Werte.

BW/4HANA – Analytics & Data Warehouse

BW/4HANA ist das modernisierte SAP Business Warehouse, komplett auf HANA native optimiert und von altem BW-Ballast befreit.

Was BW/4HANA von klassischem BW unterscheidet

MerkmalBW/4HANASAP BW 7.x (klassisch)
DatenmodellVereinfacht: aDSO als zentrales ObjektInfoProvider-Zoo: InfoCube, DSO, MultiProvider…
AggregationIn-Memory – keine vorberechneten AggregateAggregat-Tabellen notwendig für Performance
TransformationenHANA-native: SQLScript, FlowgraphsABAP-Transformationsroutinen
Daten-LayerDirekte HANA-Modellierung via CDS ViewsMehrere Schichten (PSA → DSO → InfoCube)
EchtzeitReal-Time Reporting ohne AggregataufbauAbhängig von Aggregat-Lade-Timing

Advanced DSO (aDSO) – das universelle Objekt

KERNKONZEPT BW/4HANA

Das Advanced DataStore Object (aDSO) ist das zentrale Speicherobjekt in BW/4HANA und ersetzt InfoCube, DSO und DataStore. Es kombiniert alle Speichertypen in einem einzigen, konfigurierbaren Objekt – je nach Einstellung verhält es sich als Reporting Layer, Data Mart oder Write-optimierter Store.

SAP Datasphere – der nächste Schritt

SAP Datasphere ist der Cloud-Nachfolger und ergänzt BW/4HANA um eine Business Data Fabric: es verbindet Daten aus S/4HANA, Azure, Google Cloud, und externen Quellen in einem einzigen, semantisch reichhaltigen Datenmodell – ohne physische Kopien.

Datenmodell LogistikMATDOC statt MSEG/MKPFMSEG, MKPF, MARM, MARA separat
Aggregat-TabellenKeine (Echtzeit-Aggregation in HANA)FAGLFLEXT, KNC1, LFC1 etc.

BenutzeroberflächeSAP Fiori (Web/Responsive)SAP GUI (Desktop-Client)

ErweiterbarkeitSide-by-Side auf BTP, In-App-ExtensionsUser Exits, BADIs, Enhancement Spots

Was in HANA entfallen ist (deprecated)

❌ Materialstamm-Cluster
Die MARA/MAKT/MARC/MARD-Cluster-Tabellen wurden durch ein vereinfachtes Modell ersetzt.
❌ BSEG / BKPF Aggregation
Separate FI-Dokumenttabellen ersetzt durch das Universal Journal ACDOCA.
❌ Auftragsnummernkreise
Traditionelle CO-Objekte wurden in S/4HANA vereinfacht oder neu strukturiert.
❌ Klassisches MRP
Ersetzt durch MRP Live – optimiert für HANA Parallelverarbeitung.
❌ Account-based CO-PA
Costing-based CO-PA wird schrittweise zugunsten Account-based CO-PA in ACDOCA abgelöst.
❌ Klassisches BW-Reporting
BEx Analyzer und viele BW-Objekte sind deprecated – SAC übernimmt.

Was in HANA neu hinzugekommen ist

✅ Echtzeit-Analysen
Kein nächtlicher Batch mehr – Berichte auf Live-Daten in Millisekunden.
✅ Machine Learning nativ
PAL, AutoML, Predictive direkt in der DB – ohne externe Systeme.
✅ Fiori UX
Rollenbasierte, intuitive Web-Oberflächen statt SAP GUI Transaktionen.
✅ HANA Graph/Spatial
Netzwerk- und Geodatenanalysen direkt in der Geschäfts-DB.
✅ Agile Deployments
BTP ermöglicht schnelle Cloud-Extensions ohne Core-Modifikationen.
✅ Joule (AI Copilot)
Generativer KI-Assistent direkt in S/4HANA und Fiori integriert.

🎯 Quick-Check

Welche Tabelle ersetzt in S/4HANA die getrennten FI-Tabellen BKPF und BSEG sowie CO-Tabellen?

✓ Richtig! ACDOCA ist das Universal Journal – eine Tabelle für FI, CO, ML, AA und PS-Buchungen. Das vereinfacht das Datenmodell enorm.
✗ Die Antwort ist ACDOCA – das Universal Journal. FAGLFLEXT war eine Aggregat-Tabelle (deprecated), MATDOC ist für Materialbewegungen.
Kapitel 10 · Migration

Migrationspfade von R/3 zu HANA

Es gibt verschiedene Wege von SAP ECC / R/3 nach S/4HANA – abhängig von Komplexität, Budget und gewünschtem Scope.

Die drei Migrationspfade

🔄
Greenfield (Neuimplementierung)
Was: Kompletter Neuaufbau in S/4HANA, keine Datenübernahme aus R/3.

Vorteile: Sauberes System, Best Practices, maximale HANA-Nutzung.

Nachteile: Hoher Aufwand, historische Daten bleiben im Altsystem.

Geeignet für: Organisatorische Veränderung, komplexe Altlasten.
🏗️
Brownfield (System Conversion)
Was: Technische Migration des bestehenden ECC-Systems auf S/4HANA.

Vorteile: Historische Daten bleiben erhalten, weniger Projektrisiko.

Nachteile: Altlasten (Custom Code, Z-Programme) müssen angepasst werden.

Geeignet für: Stabile ECC-Systeme, begrenzte Ressourcen.
🌿
Selective Data Transition
Was: Selektive Übernahme von Unternehmensobjekten – Mix aus Green- und Brownfield.

Vorteile: Flexibilität, nur relevante Daten migrieren, neue Strukturen möglich.

Nachteile: Komplex in der Planung und Durchführung.

Geeignet für: Fusionen, Entflechtungen, selektive Modernisierung.

Migrations-Vorgehen (Brownfield Überblick)

  1. Custom Code Analyse: SAP Readiness Check identifiziert Z-Programme, die angepasst werden müssen. Typisch: 20–40% des Custom Codes betroffen.
  2. Simplifizierungsliste prüfen: SAP liefert eine Liste aller in S/4HANA veränderten oder entfernten Objekte. Jeder Punkt muss bewertet werden.
  3. Datenbankkonvertierung: Lauf des Software Update Managers (SUM) mit dem Data Management & Landscape Transformation (DMLT) Addon.
  4. Konvertierung der DB-Tabellen: Automatische Umwandlung z.B. MSEG → MATDOC, BSEG/BKPF → ACDOCA durch Migrationsprogramme.
  5. Funktionale Aktivierung: S/4HANA-Funktionen aktivieren (Fiori, neue Prozesse), User Exits auf BADIs umstellen.
  6. Endbenutzer-Schulung: SAP Fiori statt SAP GUI – neues UX-Konzept erfordert oft vollständige Neuschulungen.
Deadline 2027/2030: SAP hat den Mainstream-Wartungsend für SAP ECC / ERP 6.0 auf Ende 2027 gesetzt (mit optionaler Extended Maintenance bis 2030). Danach gibt es nur noch Extended Customer Maintenance. Die Migration zu S/4HANA ist für die meisten Unternehmen keine Option mehr, sondern eine Pflicht.
Kapitel 11 · Praxis

Automatisierungsmöglichkeiten in SAP HANA

HANA und die BTP-Plattform bieten umfangreiche Werkzeuge für Automatisierung – von einfachen Workflows bis zu KI-gestützter Prozessoptimierung.

Prozessautomatisierung

SAP Build Process Automation (früher iRPA) RPA + Workflow
Robotic Process Automation kombiniert mit Workflow-Management. Bots können SAP-Transaktionen automatisieren (z.B. Rechnungseingang, Bestellanforderungen). Low-Code-Editor, kein Programmieraufwand für einfache Bots. Integration mit HANA-Events möglich.
SAP Workflow / BPF BTP Workflow
Business Process Flows auf BTP definieren Genehmigungsworkflows ohne Programmierung. Trigger durch S/4HANA-Events (z.B. Bestellfreigabe, Kreditlimitüberschreitung). Integration in Fiori My Inbox für Endbenutzer.
SAP Event Mesh Event-Driven
Asynchrone, ereignisgesteuerte Kommunikation zwischen Systemen. S/4HANA publiziert Business Events (z.B. "Auftrag erstellt"), die andere Systeme oder Bots abonnieren. Entkopplung und Skalierbarkeit ohne direkte System-zu-System-Aufrufe.

Intelligente Automatisierung mit KI

SAP Joule – Generativer KI-Assistent Gen AI
SAPs KI-Copilot, integriert in S/4HANA, SuccessFactors, Ariba. Nutzer stellen Fragen in natürlicher Sprache ("Zeige alle überfälligen Lieferungen"), Joule analysiert HANA-Daten und gibt strukturierte Antworten zurück. Keine SQL-Kenntnisse notwendig.
Predictive Analytics Library (PAL) ML in DB
Über 90 ML-Algorithmen direkt in HANA: Demand Forecast, Anomalie-Erkennung, Churn-Prediction. Code läuft in der Datenbank – keine Datenextraktion in externe ML-Systeme. Kann automatisch in Geschäftsprozesse integriert werden (z.B. automatische Bestellauslösung bei Nachfrage-Forecast).
SAP Document Information Extraction (DOX) AI Services
KI-basierte Dokumentenextraktion für Rechnungen, Bestellungen, Verträge. Automatische Befüllung von SAP-Feldern aus Scans oder PDFs. Reduziert manuelle Dateneingabe im Rechnungseingang massiv.

Technische Automatisierung

HANA Smart Data Integration (SDI) Datenintegration
Automatisierte Datenreplikation und -transformation aus externen Quellen in HANA. Real-Time Replication mit Change Data Capture (CDC) ohne manuelle ETL-Prozesse. Unterstützt SAP- und Non-SAP-Quellen.
Automated HANA Database Administration Self-Tuning
HANA optimiert sich selbst: automatische Memory Management, Query-Optimierung durch Adaptive Execution Engine, automatische Table Classification (hot/warm/cold Data Tiering). Der DBA-Aufwand sinkt erheblich im Vergleich zu klassischen Datenbanken.

Code Pushdown – Automatisierung auf DB-Ebene

-- ABAP Cloud: Code Pushdown statt ABAP-Loop -- ❌ Klassisches ABAP (R/3-Stil): Daten in App laden, dann verarbeiten SELECT * FROM vbap INTO TABLE @lt_items. LOOP AT lt_items INTO ls_item. ls_item-netwr = ls_item-netwr * 1.19. ENDLOOP. -- ✅ HANA-native (Pushdown): Berechnung in der DB SELECT vbeln, posnr, netwr * 1.19 AS netwr_brutto FROM vbap WHERE erdat >= @lv_date INTO TABLE @lt_items.
Ergebnis: Beim Pushdown-Ansatz werden keine Millionen Zeilen in den Applikationsserver geladen – die Berechnung erfolgt direkt in HANA mit voll parallelisierter Ausführung.
Kapitel 12 · Bewertung

Vor- und Nachteile von SAP HANA

Eine ehrliche, praxisnahe Bewertung – sowohl aus technischer als auch aus unternehmerischer Perspektive.

Technische Bewertung

✅ Vorteile (Technisch)

  • Extrem hohe Abfragegeschwindigkeit durch In-Memory und Column Store
  • OLTP und OLAP auf demselben System – keine Datenkopien notwendig
  • Native ML-Algorithmen (PAL) ohne externe ML-Infrastruktur
  • Automatisches Memory-Management und Self-Tuning
  • Graph- und Geo-Daten nativ unterstützt
  • Horizontale Skalierbarkeit für sehr große Datenmengen
  • Integrierte Volltext-Suche ohne externe Suchmaschine
  • ACID-konforme Transaktionen trotz In-Memory
  • Multi-Model: Relational, Document, Graph, Spatial in einem System

❌ Nachteile (Technisch)

  • Hohe RAM-Anforderungen – teuer bei großen Datenmengen
  • Komplexe Administration erfordert spezialisiertes Know-how
  • Delta Merge kann kurzzeitig Performance beeinflussen
  • Nicht jede Anwendung profitiert von In-Memory (kleine Transaktionssysteme)
  • Backup und Recovery komplexer als bei traditionellen DBs
  • Proprietäre Technologien (SQLScript, PAL) = Vendor Lock-in
  • Datenpersistenz erfordert sorgfältiges Savepoint-Management

Geschäftliche Bewertung

✅ Vorteile (Business)

  • Echtzeit-Reporting ohne Wartezeit auf Batch-Läufe
  • Schnellere Monats- und Jahresabschlüsse
  • Einheitliches Datenmodell (ACDOCA) reduziert Abstimmungsaufwand
  • Bessere Entscheidungsgrundlage durch Live-Daten
  • Fiori vereinfacht die Benutzerakzeptanz erheblich
  • Joule / KI-Assistenz reduziert Schulungsaufwand
  • Zukunftssicherheit (SAP ECC End of Life 2027/2030)
  • Niedrigere IT-Infrastruktur durch konsolidierte Systeme

❌ Nachteile (Business)

  • Sehr hohe Lizenz- und Implementierungskosten
  • Lange Migrationsprojekte (12–36 Monate typisch)
  • Starker Vendor Lock-in bei SAP und BTP
  • Fachkräftemangel: HANA-Spezialisten sind rar und teuer
  • Custom Code muss aufwändig angepasst werden
  • Prozessveränderungen (Best Practices) oft gegen Unternehmenskultur
  • Lernkurve für End-User trotz Fiori nicht zu unterschätzen
  • Ongoing-Kosten für Cloud-Betrieb und Updates (Public Cloud)

Wann lohnt sich SAP HANA – und wann nicht?

SzenarioHANA empfehlenswertKritisch zu hinterfragen
UnternehmensgrößeMittelstand bis Konzern (>200 User)Kleinunternehmen – TCO oft nicht wirtschaftlich
Reporting-AnforderungenKomplexe Echtzeit-Analysen nötigEinfache monatliche Reports reichen aus
DatenmengeTerabyte-Bereich, schnell wachsendStabile, kleine Datenmenge
InternationalisierungMulti-Country, Multi-Currency, Legal-EntityEinzelland, einfache Compliance

Fazit: SAP HANA ist ein leistungsfähiges, zukunftssicheres System – aber kein Allheilmittel. Der Nutzen muss gegen die erheblichen Implementierungs- und Betriebskosten abgewogen werden. Für Unternehmen, die bereits SAP ECC nutzen, ist die Migration zu S/4HANA bis 2027/2030 jedoch faktisch obligatorisch.

🎯 Frage 1 – Architektur

Was bedeutet die Abkürzung "HANA" in SAP HANA?

✓ Richtig! HANA steht für High-Performance Analytic Appliance.
✗ HANA = High-Performance Analytic Appliance. Eingeführt 2010 als In-Memory Plattform.

🎯 Frage 2 – Speicherkonzept

Welche Speichertechnik nutzt SAP HANA als primäre Datenablage?

✓ Korrekt! Daten liegen primär im RAM für maximale Geschwindigkeit, SSD dient der Persistenz.
✗ HANA hält alle operativen Daten im RAM (In-Memory). SSD dient nur der Datenpersistenz (Savepoints, Logs).

🎯 Frage 3 – Module

Welches Modul ersetzt SAP APO in S/4HANA für die Supply-Chain-Planung?

✓ Richtig! SAP IBP (Integrated Business Planning) ist der HANA-native Nachfolger von SAP APO.
✗ SAP IBP (Integrated Business Planning) löst APO ab. EWM ist Lagerverwaltung, Ariba ist Procurement.

🎯 Frage 4 – S/4HANA vs. R/3

Was ist das ACDOCA-Konzept in S/4HANA?

✓ Richtig! ACDOCA ist das Universal Journal – ersetzt BKPF, BSEG und alle CO-Einzeltabellen durch ein einheitliches Datenmodell.
✗ ACDOCA ist das Universal Journal in S/4HANA – eine Tabelle statt vieler separater FI/CO-Tabellen. Massiv vereinfachtes Finanz-Datenmodell.

🎯 Frage 5 – Automatisierung

Was versteht man unter "Code Pushdown" in SAP HANA?

✓ Richtig! Code Pushdown verlagert Logik in die DB – weniger Datentransfer, maximale Parallelisierung durch HANA.
✗ Code Pushdown bedeutet: statt Daten in den ABAP-Stack zu laden und dort zu verarbeiten, führt HANA die Logik direkt auf DB-Ebene aus.

🎯 Frage 6 – Datenmodell

Welche Migrationsstrategie wählt man, wenn man ein bestehendes ECC-System direkt zu S/4HANA konvertiert?

✓ Brownfield = technische Systemkonvertierung. Daten bleiben erhalten, das System wird direkt auf S/4HANA "upgegradet".
✗ Brownfield ist die System Conversion – das bestehende ECC-System wird technisch auf S/4HANA migriert. Greenfield = Neuaufbau.

🎯 Frage 7 – Vor- und Nachteile

Was ist ein wesentlicher Nachteil von SAP HANA aus technischer Sicht?

✓ Richtig! In-Memory bedeutet: alle Daten im RAM → teurer Arbeitsspeicher ist der primäre Kostenfaktor für On-Premise HANA-Systeme.
✗ Der größte technische Nachteil ist der hohe RAM-Bedarf. Da alle operativen Daten im Arbeitsspeicher gehalten werden, sind die Hardware-Kosten erheblich.

🎯 Frage 8 – Cloud & BTP

Was ist SAP Joule?

✓ Richtig! Joule ist SAPs KI-Assistent – man stellt Fragen in natürlicher Sprache und bekommt Analysen und Aktionen direkt in SAP zurück.
✗ Joule ist SAPs generativer KI-Copilot (ähnlich wie Microsoft Copilot), der in S/4HANA, SuccessFactors und Ariba integriert ist.
